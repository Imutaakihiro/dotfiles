## 正規化の目的

正規化とは、データの**冗長性を排除**し、**更新異常を防ぐ**ための設計プロセス。

### 更新異常の3種類

| 異常の種類 | 説明 | 例 |
|---|---|---|
| 挿入異常 | 主キーの一部が NULL になる場合に挿入できない | 学生のいないコースを登録できない |
| 削除異常 | 1件削除すると必要な情報まで消える | 最後の受講生を削除するとコース情報が消える |
| 更新異常 | 同じデータが複数箇所にあり、更新漏れが起きる | 講師の名前変更が一部のレコードに反映されない |

## 第1正規形 (1NF)

**条件：** セルの値が1つの値（原子値）であること

### 違反例

```
受注テーブル
| 受注ID | 顧客名 | 商品リスト              |
|--------|--------|------------------------|
| 1      | 田中   | りんご, バナナ, みかん  |
```

### 1NF に変換

```
受注テーブル
| 受注ID | 顧客名 |

受注明細テーブル
| 受注ID | 明細番号 | 商品名   |
|--------|----------|---------|
| 1      | 1        | りんご  |
| 1      | 2        | バナナ  |
| 1      | 3        | みかん  |
```

**NG パターン：**
- カンマ区切りの値（`"りんご,バナナ"`）
- 配列・JSON 列（原則として。ただし BigQuery では意図的に使うケースあり）
- 繰り返しグループ（`商品1`, `商品2`, `商品3` という列名）

## 第2正規形 (2NF)

**条件：** 1NF を満たし、かつ**部分関数従属がない**こと
（複合主キーの一部にのみ従属する属性を排除する）

### 違反例（複合主キー: 受注ID + 商品ID）

```
受注明細テーブル
| 受注ID | 商品ID | 商品名   | 数量 | 単価   |
| ← 受注IDのみに従属        →|      |← 商品IDのみに従属 →|
```

`商品名` と `単価` は `商品ID` だけで決まるため部分従属。

### 2NF に変換

```
受注明細テーブル
| 受注ID | 商品ID | 数量 |   ← 複合キー全体に従属

商品テーブル
| 商品ID | 商品名 | 単価 |   ← 商品IDのみに従属
```

## 第3正規形 (3NF)

**条件：** 2NF を満たし、かつ**推移関数従属がない**こと
（非キー属性が別の非キー属性を経由してキーに従属する構造を排除する）

### 違反例

```
社員テーブル
| 社員ID | 社員名 | 部署ID | 部署名 | 部署所在地 |
         主キー → 部署ID → 部署名, 部署所在地（推移従属）
```

`部署名` と `部署所在地` は `部署ID` に従属し、さらに `部署ID` は `社員ID` に従属。

### 3NF に変換

```
社員テーブル
| 社員ID | 社員名 | 部署ID |

部署テーブル
| 部署ID | 部署名 | 部署所在地 |
```

## ボイス・コッド正規形 (BCNF)

**条件：** 3NF を満たし、かつ**すべての関数従属の決定子がスーパーキー**であること
（3NF より少し厳しい。複数の候補キーが重なる場合に問題が起きる）

### 違反例

```
講座割当テーブル
候補キー1: (学生ID, 講座ID)
候補キー2: (学生ID, 講師ID)

| 学生ID | 講座ID | 講師ID |
```

`講師ID → 講座ID` という従属があり、`講師ID` は候補キーの一部だがスーパーキーではない。

### BCNF に変換（必ずしも無損失分解できるとは限らない）

```
講師担当テーブル
| 講師ID | 講座ID |

受講テーブル
| 学生ID | 講師ID |
```

**注意：** BCNF への変換で関数従属が保存できなくなる場合は 3NF に留める判断も有効。

## 非正規化の判断基準

正規化は原則だが、**意図的な非正規化**が有効な場合もある。

### 非正規化してよい条件

1. **読み取りが圧倒的に多い**（DWH・分析用テーブル）
2. **結合コストがボトルネックになっている**（実測で確認済み）
3. **更新頻度が極めて低い**（変更の可能性が低いマスタ系データ）

### 非正規化してはいけない条件

- トランザクション処理が多い業務DB
- 更新が頻繁に発生するデータ
- 整合性が必須のデータ（残高、在庫等）

### BigQuery での非正規化

BigQuery は結合コストが高いため、**ネストされた繰り返しフィールド（ARRAY/STRUCT）による非正規化**が推奨される。
→ 詳細は `references/bigquery-specifics.md` を参照。

## 実践的な正規化の進め方

1. **エンティティの識別** — 何を管理したいかを動詞（注文する・在庫を持つ）で考える
2. **属性の洗い出し** — 各エンティティが持つべき情報を列挙する
3. **主キーの決定** — 自然キーか代理キーかを判断する
4. **従属関係の確認** — 「AはBによって一意に決まるか？」を問い続ける
5. **正規形の確認** — 1NF → 2NF → 3NF の順に確認する
6. **逆正規化の検討** — パフォーマンス要件・更新頻度を踏まえて判断する

### 自然キー vs 代理キーの判断

| 観点 | 自然キー | 代理キー（サロゲートキー）|
|---|---|---|
| 意味のある値 | ある | ない（採番のみ） |
| 変更リスク | 高い（業務ルール変更で変わる） | なし |
| 結合のしやすさ | 可読性が高い | JOIN が増える可能性 |
| 推奨シーン | コード系マスタ（ISO国コード等） | トランザクション系・変更可能な値 |
